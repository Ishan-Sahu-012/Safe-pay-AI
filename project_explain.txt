1. PROJECT OVERVIEW
- Project name: SafePay AI
- Main purpose: Provide an AI-assisted fraud detection backend for QR/UPI scams, SMS scam messages, and URL phishing, with user authentication and real-time alerting.
- Real-world problem solved: Detect fraudulent payment requests, scam messages, and suspicious URLs to protect users from financial and social-engineering attacks.
- Core features:
  - JWT-based authentication and registration
  - UPI/QR fraud scanning and analysis
  - SMS/text scam detection
  - WebSocket threat alert broadcasting
  - Health monitoring and system diagnostics
  - SQL database with user history and risk registries
  - Hybrid ML and rule-based fraud scoring
  - QR image processing and decoding
  - Dashboard/static frontend serving

4. EXECUTION FLOW
Explain complete end-to-end flow:
App start
→ User action
→ Frontend
→ API call
→ Backend processing
→ Database
→ ML models
→ Response
→ UI output

Detailed step-by-step flowchart:
1. Start App:
   - `run.py` executes.
   - Loads `app.config.settings`, `app.utils.logger`.
   - Checks database connectivity via `app.database.db.engine`.
   - Checks ML health by calling `app.ml.inference.upi_predictor.model_health` and `app.ml.inference.text_predictor.model_health`.
   - Launches Uvicorn with `app.main:app`.
2. App initialization in `app.main`:
   - Create FastAPI app with CORS, GZip, custom middleware.
   - Mount `/static` directory.
   - Register auth, fraud, qr, sms, health, ws routers.
   - On startup lifespan: create DB tables via SQLAlchemy metadata and seed registry tables if empty.
   - Log ML health and startup.
3. User action / Frontend request:
   - User opens dashboard at `/dashboard` or Chrome extension interacts with endpoints.
   - Dashboard or extension sends HTTP requests to `/auth/login`, `/fraud/scan-text`, `/fraud/scan-qr`, `/sms/scan`, etc.
4. API call interception:
   - `LoggingMiddleware` logs request details, sets request ID, times execution.
   - `AuthMiddleware` checks exclusions. For protected routes, it reads `Authorization: Bearer <token>` and validates JWT.
   - If valid, request.state.user is populated.
5. Backend route processing:
   - Route-specific request model validation occurs via Pydantic.
   - Dependencies inject `db` session and `current_user` if needed.
   - `auth_routes` uses `get_db()` for login/register and `get_current_user()` for protected endpoints.
   - `fraud_routes` and `sms_routes` call `app.services.fraud_detection_service` functions.
   - `qr_routes` decodes uploaded image via OpenCV/PIL/pyzbar, then calls `detect_qr_fraud` and saves QR history.
6. ML and fraud engine processing:
   - `fraud_detection_service.detect_qr_fraud` / `detect_text_fraud` call `execute_fraud_pipeline`.
   - `pipeline.py` extracts text features, analyzes domains, UPI IDs, sender mismatches, social engineering cues.
   - It queries ML inference wrappers: `predict_upi_fraud`, `predict_text_scam`.
   - `predict_upi_fraud` loads model `app/ml/models/model1_upi.pkl`, extracts UPI features, predicts probability, applies rule scoring.
   - `predict_text_scam` loads text model `app/ml/models/model2_text.pkl`, cleans text, extracts text features and computes hybrid score.
   - URL analysis may call `app.ml.inference.url_predictor`.
   - `RiskScoringEngine` aggregates all signal scores and normalizes to 0–100.
   - `ExplainableDecisionEngine` maps final score to `SAFE`, `LOW`, `MEDIUM`, `HIGH`, `CRITICAL`.
7. Database read/write:
   - User data stored in `users` table.
   - Scan results written to `scan_history`, `sms_scan_history`, `qr_scan_history`.
   - Reports update `upi_merchant_registry` or log to `communication_intercept_logs`.
   - `health_routes` query DB for `SELECT 1`.
   - Fraud URL analysis reads `URLSecurityRegistry` and caches registry results.
8. Response and UI output:
   - Routes assemble JSON responses with `success`, `analysis`, `risk_score`, `risk_level`, `reasons`.
   - `LoggingMiddleware` appends response headers and logs latency/status.
   - UI dashboard or extension displays risk and threat alerts.
9. Real-time alerting:
   - If risk score >= 45, routes create async broadcasts via `ws_manager.broadcast`.
   - Active WebSocket clients connected to `/ws/alerts?token=<jwt>` receive JSON alert messages.

7. API ANALYSIS
List all APIs/endpoints:
- `GET /` → root; success message, version, docs/dashboard links.
- `GET /dashboard` → serve static HTML dashboard.
- `GET /system/info` → app/system metadata.
- `GET /ping` → readiness.
- `POST /auth/register` → body `name,email,phone,password`; registers user, stores hashed password, returns JWT.
- `POST /auth/login` → body `phone/email,password`; verifies credentials, returns JWT.
- `GET /auth/profile` → JWT protected; returns user details.
- `GET /auth/me` → alias of profile.
- `POST /auth/refresh-token` → JWT protected; issues new token.
- `POST /auth/logout` → placeholder, frontend-managed.
- `POST /fraud/scan-qr` → JWT protected; body `upi_id,amount,merchant_name?,ocr_text?`; runs QR fraud detection, saves scan history, broadcasts if risk >=45.
- `POST /fraud/scan-text` → JWT protected; body `text,sender?,ocr_text?`; runs text fraud detection, saves scan history, broadcasts if risk >=45.
- `POST /fraud/bulk-scan` → JWT protected; list of QR scan requests; returns batch results; broadcasts high-risk.
- `GET /fraud/history` → JWT protected; returns user scan history from `ScanHistory`.
- `POST /fraud/report-upi` → JWT protected; body params `upi_id,reason`; logs report and updates `UPIMerchantRegistry`.
- `POST /qr/scan` or likely `POST /qr/...` in route (not fully read due file truncation) for image upload and decode.
- `GET /qr/health` likely exists from route or `health_routes` config? Actually `/health/ml` and other health.
- `POST /sms/scan` → JWT protected; body `message,sender?,ocr_text?`; runs SMS scan, stores history, logs intercept.
- `POST /sms/bulk-scan` → JWT protected; body `messages: [string]`; batch text scan.
- `GET /sms/history` → JWT protected; returns SMS scan history.
- `POST /sms/report` → JWT protected; report scam SMS.
- `GET /sms/health` → returns `service: SMS Scam Detection running`.
- `GET /health/` → basic health.
- `GET /health/database` → DB connectivity/latency.
- `GET /health/ml` → model file existence.
- `GET /health/system` → CPU/memory/disk/uptime.
- `GET /ws/alerts` (WebSocket) → accepts JWT query param `token`, authenticates, keeps connection alive.

8. DATABASE ANALYSIS
Explain:
- tables:
  - `users`: registered users.
  - `scan_history`: general scan records for user scans.
  - `sms_scan_history`: message scan records.
  - `qr_scan_history`: decoded QR scan records.
  - `messages`: imported message corpus for ML/training.
  - `upi_merchant_registry`: merchant fraud registry for UPI IDs.
  - `url_security_registry`: registry of URL safety statuses.
  - `communication_intercept_logs`: intercepted message logs.
- schema:
  - `users`: id, name, email unique, phone unique, password, role, is_active, created_at.
  - `scan_history`: user_id FK, scan_type, input_value, risk_score, risk_level, status, created_at.
  - `sms_scan_history`: user_id FK, sender, message, risk_score, risk_level, status, created_at.
  - `qr_scan_history`: user_id FK, upi_id, merchant_name, amount, raw_qr, risk_score, risk_level, status, created_at.
  - `messages`: message id, label, text, length, scam category, risk score/level.
  - `upi_merchant_registry`: merchant_id PK, upi_id unique, merchant_name, verification flag, report_count, avg transaction.
  - `url_security_registry`: url_id PK, raw_target_url, associated_brand, safety_status, tld_extension.
  - `communication_intercept_logs`: message_id PK, sender_address, message_body, timestamp_received, reported_scam_category.

  - `User` has one-to-many with `ScanHistory`, `SMSScanHistory`, `QRScanHistory`.
